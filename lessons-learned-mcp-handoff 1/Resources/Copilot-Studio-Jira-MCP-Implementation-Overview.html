01 The problem we solved

The Quality team wants Copilot Studio agents to log new lessons-learned entries into an on-prem Jira system. Reads are already handled via a scheduled export — the blocker is secure write-back: the prebuilt Jira connector is read-only, and on-prem Jira cannot be exposed to the internet.

🔒 Non-negotiable

On-prem Jira stays private. No inbound internet exposure — enforced by design, not policy.

✍️ The gap

Agents need to create issues. The out-of-the-box connector only supports GET.

🧩 The approach

An Azure-hosted MCP server as the secure middle tier — with a seam to plug in real Jira later.

02 Architecture — authentication & write-back flow

Component-level view of the secured call path across the Microsoft Cloud, the Azure landing zone (private topology shown), and Harman on-premises. Numbered steps trace a single write-back request end to end.

1Business user asks the agent to log a lesson learned.
2Agent acquires an access token from Microsoft Entra ID (OAuth 2.0 + PKCE).
3Agent calls the MCP endpoint over HTTPS with the Bearer JWT.
4API Management validates the JWT, applies rate limits, and forwards to the server, which re-validates the token.
5Server retrieves the Jira secret from Key Vault via Managed Identity over a Private Endpoint.
6Server creates the issue in Jira Data Center privately over the existing ExpressRoute (write-back).
7The structured response flows back through the gateway to the agent.
·Telemetry (traces & logs) streams continuously to Application Insights.

Icons follow the official Microsoft Azure architecture icon style. Private topology shown; in public mode the agent reaches the Container App over a secured public HTTPS endpoint without API Management or private endpoints.

This deployment uses private mode from the start — the ingress.external:true setting that makes the internal Container App reachable through APIM is verified in Deployment step 6.

03b Private connectivity — two approaches

Copilot Studio runs in Power Platform (Microsoft SaaS), so it calls the MCP endpoint from outside the customer network — the classic “works internally, fails externally” gap. “Private” does not mean “no ingress”; it means the backend and data path are private, and the only reachable surface is a hardened, governed gateway. There are two enterprise-grade ways to close that gap.

Approach A — Governed public front door (this build)Approach B — Fully private
How Copilot Studio reaches itPublic APIM gateway (VNet-injected, External mode)Power Platform VNet integration → internal APIM (no public IP)
Public surfaceOne authenticated, WAF-able endpoint, restrictable to Power Platform IP rangesNone
Security controlsEntra JWT + rate-limit at APIM, defense-in-depth at the app; optional Front Door / App Gateway WAFSame, plus zero internet exposure
PrerequisitesNone beyond AzureManaged Environments (Power Platform premium), region support, network-admin setup — see VNet support for agent calls to private endpoints
Best forFast, standard AI-gateway pattern; demo → productionRegulated “nothing internet-facing” mandates

Why A first: it is the common production shape for an AI gateway, needs no Power Platform networking licensing, and is quick to stand up — Copilot Studio hits the public APIM gateway, APIM validates the Entra token and rate-limits, then forwards over the VNet to the internal Container App. The backend, Key Vault, registry, and on-prem Jira leg stay fully private. Approach B is the enterprise end-state once the customer has (or licenses) Managed Environments; A can run as the interim while that is provisioned.

Fit to the customer’s current state: their “works internally, fails externally” is precisely this gateway gap. Two ways to adopt: (1) re-host the MCP server in Azure (this build) so Azure is the governed layer, or (2) front their existing internal server with the same gateway. Either way, do not simply open a firewall port — expose it through the governed gateway (Entra JWT + WAF / rate-limit), never raw.

This build implements Approach A: an External, VNet-injected APIM (with the required Standard public IP and NSG) in front of an internal Container App. The step-by-step setup — including the exact settings to verify — is in the Deployment Walkthrough tab.

1

Prepare your tooling

In plain terms · install the prerequisites and sign in to Azure.

# prerequisites: Node 20, Docker, Azure CLI, azd

2

Register the Azure resource providers

In plain terms · switch on the Azure services this deployment uses, once per subscription.

Because you deploy the private topology from the start, the Container Apps environment is created VNet-injected (internal) on the very first azd up. That needs both Microsoft.App and Microsoft.ContainerService registered, or provisioning fails with SubscriptionIsNotRegistered. Getting this right on day one means the environment is born internal — VNet injection is immutable, so there is no later rebuild.

az provider register --namespace Microsoft.App
az provider register --namespace Microsoft.ContainerService
# APIM, Key Vault, ACR and networking are registered automatically by azd
3

Register the API identity & expose the OAuth scope

In plain terms · create the Entra identity that controls who may call the endpoint, and publish the permission Copilot Studio will request.

Create an Entra ID app registration to protect the MCP endpoint, set the Application ID URI (audience) to api://<client-id>, then expose a delegated scope named mcp.tools that Copilot Studio requests during OAuth consent. The whole flow is scriptable — no portal clicks required. Record the tenant id and client id for step 4.

# 1) Create the app registration and capture its client id
$appId = az ad app create --display-name "Lessons-Learned MCP API" --query appId -o tsv

# 2) Set the Application ID URI (the token audience)
az ad app update --id $appId --identifier-uris "api://$appId"

# 3) Expose the delegated scope 'mcp.tools' via Microsoft Graph.
#    Save the JSON body below as expose-scope.json first (generate a
#    fresh GUID for "id" with [guid]::NewGuid()).
$oid = az ad app show --id $appId --query id -o tsv
az rest --method PATCH `
  --uri "https://graph.microsoft.com/v1.0/applications/$oid" `
  --headers "Content-Type=application/json" `
  --body "@expose-scope.json"

# 4) Verify the scope is published and enabled
az ad app show --id $appId --query "api.oauth2PermissionScopes[].{value:value,enabled:isEnabled}" -o json

# 5) Force v2.0 access tokens. Without this, Entra issues v1.0 tokens
#    (issuer https://sts.windows.net/...) which the server rejects.
az rest --method PATCH `
  --uri "https://graph.microsoft.com/v1.0/applications/$oid" `
  --headers "Content-Type=application/json" `
  --body '{\"api\":{\"requestedAccessTokenVersion\":2}}'

expose-scope.json — the scope definition sent to Graph:

{
  "api": {
    "oauth2PermissionScopes": [
      {
        "id": "<new-guid>",
        "value": "mcp.tools",
        "type": "User",
        "isEnabled": true,
        "adminConsentDisplayName": "Invoke Lessons Learned MCP tools",
        "adminConsentDescription": "Allows the client (Copilot Studio) to call the Lessons Learned MCP server tools on behalf of the signed-in user.",
        "userConsentDisplayName": "Invoke Lessons Learned MCP tools",
        "userConsentDescription": "Allows this app to call the Lessons Learned MCP server tools on your behalf."
      }
    ]
  }
}

Copilot Studio requests this scope as api://<client-id>/mcp.tools when it authenticates to the endpoint. The server validates the token's issuer (must be the v2.0 endpoint https://login.microsoftonline.com/<tenant>/v2.0) and audience. Because a v2.0 token's aud claim is the bare client-id GUID (a v1.0 token uses api://<client-id>), the server accepts both forms — so no audience change is needed when you switch token versions.

Provision and deploy

In plain terms · one command builds the image and stands up the entire private topology.

This builds the container, pushes it to the registry, and provisions everything — VNet, internal Container App, APIM, Key Vault, and private endpoints. Because APIM is VNet-injected, allow roughly 45–90 minutes for this first run.

6

Verify the private-mode settings (three quick checks)

In plain terms · confirm the handful of settings that make private mode actually work — all baked into this build, but worth verifying.

These are the exact settings that, if wrong, make APIM return errors even though the app is healthy. This build sets them correctly in Bicep; verify them (and re-apply if you ever hand-edit the infra).

6a · Container App ingress is external: true. The single most important setting. In a VNet-injected internal environment the app must use external:true so the environment’s envoy routes it at <app>.<defaultDomain>. It stays fully VNet-private because the environment is internal (no public IP). With external:false every call returns Error 404 — This Container App is stopped or does not exist, even though the app is Ready.

# verify (expect: true)
az containerapp show -n <app> -g <rg> --query "properties.configuration.ingress.external"
# fix if needed
az containerapp ingress update -n <app> -g <rg> --type external

6b · APIM “subscription required” is OFF. Copilot Studio sends only the Entra OAuth bearer token, not an APIM subscription key, so the JWT policy is the auth gate. If a call returns 401 "missing subscription key", disable it:

az apim api update --service-name <apim> -g <rg> --api-id mcp --subscription-required false

6c · APIM backend points at the non-.internal. FQDN. Its serviceUrl must be https://<app>.<defaultDomain> (the address external:true serves) — not the .internal. form.

az apim api show --service-name <apim> -g <rg> --api-id mcp --query serviceUrl -o tsv

Two more requirements are already handled in this build’s code and infra: the APIM JWT policy accepts both audience forms (api://<client-id> and the bare GUID), and the private DNS zone is derived from the environment’s own defaultDomain + staticIp so it self-corrects on any rebuild. If you bring your own MCP image, also apply the two transport requirements in Path 2 below.

8a · Create a client secret on the app registration from step 3 (record the value immediately — it is shown only once):

8b · Add the tool. In your agent: Tools → Add a tool → New tool → Model Context Protocol. Provide the endpoint URL (the MCP_ENDPOINT_URL from step 7). Under Authentication choose OAuth 2.0, then Type → Manual.

8c · Fill in the OAuth fields:

FieldValue

P1 Path 1 · Swap to your Jira in place

Keep this build; replace the representative store with calls to the real Jira. It is a config + one-file change, never a re-architecture.

1

Implement the provider seam

In plain terms · fill in four methods that call the Jira REST API.

Open src/providers/jiraDataCenterProvider.ts. It implements the same ILessonsProvider interface as the placeholder, with four stubbed methods marked TODO(customer). Implement each against the Jira Data Center/Server REST API (attach whichever auth header your chosen scheme requires — see step 2):

MethodJira REST call
get(issueKey)GET /rest/api/2/issue/{issueKey} → map fields to LessonLearned
search(query)POST /rest/api/2/search with JQL scoped to your project key
create(input)POST /rest/api/2/issue (fields.project.key, issuetype, summary, description) — the write-back path
update(issueKey,input)PUT /rest/api/2/issue/{issueKey}, then re-fetch and map

Reference implementation (PAT scheme) — the seam is a thin REST client: build the auth header from the injected secret, then map fields both ways.

// src/providers/jiraDataCenterProvider.ts
const base    = process.env.JIRA_BASE_URL;      // https://jira.internal.contoso.com
const project = process.env.JIRA_PROJECT_KEY;   // e.g. LL
const authHeader = { Authorization: `Bearer ${process.env.JIRA_PAT}` };  // PAT from Key Vault

async create(input: LessonInput): Promise<LessonLearned> {
  const res = await fetch(`${base}/rest/api/2/issue`, {
    method: 'POST',
    headers: { ...authHeader, 'Content-Type': 'application/json' },
    body: JSON.stringify({ fields: {
      project:   { key: project },
      issuetype: { name: 'Task' },
      summary:     input.title,
      description: input.body
    }})
  });
  if (!res.ok) throw new Error(`Jira create failed: ${res.status}`);
  const issue = await res.json();   // { key: "LL-123", ... }
  return this.get(issue.key);       // re-fetch, then map to LessonLearned
}

Nothing else changes — transport, OAuth, hosting, and observability are identical. The tool schemas Copilot Studio already sees stay the same.

2

Choose an auth scheme & store the credential in Key Vault

In plain terms · pick how the server proves who it is to Jira — the credential never lives in code or config.

A Personal Access Token is the recommended default, but it is not the only option. Jira Data Center/Server supports several REST auth schemes, and the provider seam is agnostic — it is ultimately just an Authorization header, so any of these fit behind the same code:

SchemeHeaderNotes
Personal Access Token (PAT)Authorization: Bearer <token>Recommended. Jira DC/Server 8.14+. Tied to a service account, scoped, revocable, no password.
Basic auth (service account)Authorization: Basic base64(user:pass)Works on-prem. Simpler, but stores a real password and inherits rotation/lockout policy.
OAuth 1.0a (Application Link)RSA-SHA1 signed requestMore setup (key exchange, consumer config). Common where PATs are disallowed by policy.
OAuth 2.0 / OIDC (external IdP)Authorization: Bearer <token>Only if Jira DC is fronted by an IdP; uncommon for server-to-server REST.

For the recommended PAT path, create the token against a dedicated Jira service account with least-privilege project permissions, then store it in the Key Vault the deployment created as the secret named jira-pat (the infra already references this name and injects it as an env var via the managed identity):

az keyvault secret set `
  --vault-name <your-key-vault> `
  --name jira-pat `
  --value <jira-personal-access-token>

Switching schemes changes only two things: how the Authorization header is built in the provider, and the secret(s) you store here (e.g. a user:password pair for Basic auth). The transport, hosting, and Copilot Studio/OAuth wiring are untouched.

3

Point the app at Jira

In plain terms · set the provider switch and Jira coordinates, then redeploy. You are already in private mode, so there is no topology change.

Set the provider to jira and supply the Jira coordinates. The app already runs in the private topology, so it reaches the on-prem server over your existing ExpressRoute / private DNS:

azd env set PROVIDER_MODE jira
azd env set JIRA_BASE_URL https://jira.internal.contoso.com
azd env set JIRA_PROJECT_KEY LL
azd deploy

Network prerequisites (one-time, typically your platform/network team) — required before PROVIDER_MODE=jira will work:

  • Routing: the Container Apps subnet (snet-container-apps) must reach the on-prem Jira network over ExpressRoute (or site-to-site VPN). Confirm a route to the Jira subnet exists.
  • DNS: the app must resolve JIRA_BASE_URL to its on-prem IP. Point the VNet at your DNS — custom DNS servers or an Azure Private DNS Resolver that forwards the on-prem zone — then verify resolution from inside the VNet.
  • Egress on 443: allow outbound HTTPS from the app subnet to the Jira host. If a firewall/NSG restricts egress, add an allow rule for the Jira host and port.
  • TLS trust: if Jira presents an internal (private CA) certificate, ensure that CA is trusted by the container — bundle the CA in the image or provide it via NODE_EXTRA_CA_CERTS.

Fast pre-flight from a jump box in the VNet: nslookup jira.internal.contoso.com then curl -I https://jira.internal.contoso.com — both must succeed before the app can talk to Jira.

P2 Path 2 · Bring your own MCP server

Keep everything you built — the Entra identity, private VNet, APIM gateway, Key Vault, and Copilot Studio connection — and run your MCP server in place of the representative one. Only the container image changes.

1

Containerize your MCP server

In plain terms · package your server so this platform can run it.

Provide a container that listens on the port in $PORT (default 8080), exposes /health and /ready for the platform probes, and serves the MCP Streamable HTTP transport at /mcp. Validating the Entra JWT in-app (same tenant + audience) is recommended defense-in-depth on top of the APIM gateway check.

2

Meet the two “MCP behind APIM” requirements

In plain terms · two transport settings every MCP server needs to work through APIM.

These are the two things that bit this build and will bite any MCP server placed behind APIM:

  1. Return single JSON responses, not SSE. APIM buffers Server-Sent Events, which breaks the handshake. On the MCP StreamableHTTPServerTransport set enableJsonResponse: true so it replies with one application/json body.
  2. Handle GET and DELETE on /mcp. The Streamable HTTP transport uses GET (event stream) and DELETE (session end) in addition to POST. APIM must expose all three operations on /mcp — this build already defines them; keep them if you edit the API.
3

Swap in your image

In plain terms · tell the deployment to run your container instead of the sample.

Keep the mcp service in azure.yaml (so the managed identity, APIM backend, and Key Vault bindings stay wired) and point it at your code or a prebuilt image:

🔧 Troubleshooting OAuth

The most common first-connection issue and its resolution, captured from a live bring-up.

Connector request failed — "Invalid bearer token" (JSON-RPC code -32001)

In plain terms · sign-in succeeded, but the server rejected the token it was handed.

The Copilot Studio connection shows connected (green), yet the tool list stays empty and View details shows {"error":{"code":-32001,"message":"Invalid bearer token."}}. This is the resource server rejecting the token — almost always a token-version mismatch. The server only accepts Entra v2.0 tokens, but a new app registration issues v1.0 tokens by default.

Fix — three parts:

  1. Set the app to issue v2.0 tokens (step 2, command 5): requestedAccessTokenVersion = 2.
  2. Ensure the server accepts both audience forms — the api://<client-id> URI (v1.0) and the bare <client-id> GUID (v2.0). This build already does.
  3. In Copilot Studio, disconnect and reconnect the tool's connection so it fetches a fresh v2.0 token — the cached v1.0 token is not refreshed automatically.

After reconnecting, the Tools list populates and create/get calls succeed. To confirm the token shape, the server validates issuer https://login.microsoftonline.com/<tenant>/v2.0 and audience — a v1.0 token (issuer https://sts.windows.net/<tenant>/) fails the issuer check.
